# 10. UI Workspace Architecture

## Назначение

Этот документ описывает целевую UX-архитектуру пользовательского интерфейса для PoV Generator.

Документ отвечает на вопросы:

- как пользователь видит проект целиком;
- какие экраны существуют;
- как они связаны друг с другом;
- какие данные нужны каждому экрану;
- как UI должен использовать уже реализованный `M9` API;
- какие серверные расширения понадобятся на `M10`.

Это **не спецификация визуального стиля**. Визуальный язык и токены вынесены в [11_ui_design_system.md](F:\0work\python\PoV-generator\specs\11_ui_design_system.md).

---

## Главная продуктовая идея

Интерфейс не должен выглядеть как:

- лог-система;
- агентная IDE;
- таблица внутренних сущностей;
- “дашборд ради дашборда”.

Интерфейс должен выглядеть как **рабочее пространство проекта**, где пользователь за 10–15 секунд понимает:

1. что это за проект;
2. где он сейчас находится;
3. что уже произошло;
4. что мешает двигаться дальше;
5. что нужно сделать прямо сейчас.

Главный экран должен отвечать на эти вопросы без перехода по вторичным разделам.

---

## Роли

### Основной пользователь

Это заказчик, аналитик или владелец кейса.

Ему нужно:

- видеть бизнес-смысл проекта;
- отвечать на уточнения;
- читать артефакты;
- понимать блокировки;
- принимать или отклонять результат.

### Оператор / внутренний аналитик

Это человек, который сопровождает работу системы.

Ему нужно:

- видеть детальный путь выполнения;
- разбирать ошибки;
- смотреть traces и validation;
- вручную запускать команды;
- управлять переходами в пограничных кейсах.

### Архитектурное правило

Интерфейс один, но имеет **две глубины просмотра**:

- business layer;
- operator layer.

Business layer доступен сразу.
Operator layer открывается как “углубление в детали”, а не как отдельный продукт.

---

## Информационная архитектура

UI строится вокруг двух уровней:

1. `Project Index`
2. `Project Workspace`

### 1. Project Index

Это список кейсов.

Отвечает на вопросы:

- какие проекты существуют;
- какие из них требуют внимания;
- какие завершены;
- где сейчас есть блокировки.

### 2. Project Workspace

Это главный уровень продукта.

Внутри него:

- `Обзор`
- `Артефакты`
- `Ход выполнения`
- `Состояние`
- `Замечания`
- `Технические детали`

Пользователь должен в основном жить в `Обзоре`.
Остальные разделы нужны для drill-down.

---

## Главный экран проекта

Главный экран проекта называется **Workspace Overview**.

### Структура сверху вниз

1. `Project Header`
2. `Journey Strip`
3. `Операционный центр`
4. `Ключевые артефакты`
5. `Контекстные детали`

---

## 1. Project Header

Верхний блок.

Показывает:

- название проекта;
- короткий исходный запрос;
- статус проекта человеческим языком;
- время последнего обновления;
- активные domain packs;
- быстрые вторичные действия.

Задача блока:

- сразу объяснить, что за кейс открыт;
- убрать ощущение “безличного объекта из системы”.

---

## 2. Journey Strip

Горизонтальная линия пути проекта.

Показывает:

- этапы composed recipe;
- выполненные шаги;
- текущий шаг;
- ожидающие шаги;
- заблокированные шаги;
- шаги, добавленные domain packs.

### UX-правила

- это не технический DAG;
- это читаемый маршрут проекта;
- шаги должны называться на человеческом языке;
- текущий шаг должен быть визуально доминирующим;
- шаги domain pack должны быть видны, но не ломать общую структуру.

### Действия

Нажатие на шаг открывает детальную карточку шага внутри раздела `Ход выполнения`.

---

## 3. Операционный центр

Это главный блок всего интерфейса.

Он объединяет:

- текущую ситуацию;
- главное следующее действие;
- историю происходящего.

### 3.1. Текущая ситуация

Блок из 2–4 строк, который отвечает на вопрос:

- что происходит сейчас.

Он должен показывать:

- headline;
- короткое summary;
- признак блокировки;
- кто сейчас должен действовать.

Источники данных:

- `situation`
- `review`
- `state`

### 3.2. Primary Action

На экране должен быть **ровно один главный action**.

Примеры:

- `Открыть замечания`
- `Ответить на уточнения`
- `Посмотреть ТЗ`
- `Продолжить выполнение`
- `Запустить доработку`

Правило:

- frontend не вычисляет главный action сам;
- backend отдаёт его как отдельную сущность;
- UI отвечает только за отображение и подтверждение действия.

### 3.3. Операционная лента

Это человекочитаемая хронология проекта.

Она показывает:

- ключевые изменения понимания проекта;
- выпуск важных артефактов;
- изменения состояния процесса;
- блокировки, замечания и escalations;
- важные ручные действия пользователя или оператора.

Это **не raw log**.

Каждое событие должно быть:

- лаконичным;
- читаемым;
- осмысленным;
- кликабельным.

#### Примеры корректных событий

- “Система уточнила бизнес-цель”
- “Собраны ключевые user story”
- “Подготовлен черновик ТЗ”
- “Ревью выявило 4 замечания”
- “Процесс остановлен: требуется ответ пользователя”

#### Примеры некорректных событий

- `task transitioned to in_progress`
- `validation_run failed`
- `patch applied`

#### Правила ленты

- показывать только смысловые события;
- группировать плотные серии системных действий в один смысловой блок;
- выделять проблемные события;
- позволять открыть связанную сущность из карточки события.

---

## 4. Ключевые артефакты

Это не timeline и не лог.
Это компактная библиотека результатов проекта.

Показываются только действительно важные артефакты:

- уточнение цели;
- карта user story;
- анализ альтернатив;
- черновик ТЗ;
- отчёт ревью;
- UI outline или другие доменные артефакты, если включены domain packs.

Для каждого артефакта показываются:

- тип;
- заголовок;
- статус;
- время создания;
- наличие замечаний;
- действие `Открыть`.

Правило:

- список должен быть компактным и сканируемым;
- не показывать внутренние вспомогательные артефакты на главном экране.

---

## 5. Контекстные детали

Нижняя или правая часть overview показывает краткие сводки:

- состояние проекта;
- readiness;
- активные gaps;
- краткий review summary;
- последние технические сигналы.

Это именно summary.
Полные детали открываются во вторичных разделах.

---

## Вторичные разделы

### Обзор

Главный экран.
Используется почти всегда.

### Артефакты

Список и чтение проектных артефактов.

Внутри:

- reader-first view;
- structured JSON как вторичная вкладка;
- provenance;
- связанные проверки.

### Ход выполнения

Раздел для понимания маршрута проекта.

Внутри:

- composed recipe;
- статусы шагов;
- происхождение шагов:
  - base recipe;
  - recipe fragment;
  - domain pack;
- прогресс;
- точки остановки.

### Состояние

Структурированное представление `ProblemState`.

Внутри:

- цель;
- active gaps;
- readiness;
- assumptions;
- facts;
- domain packs;
- recipe composition.

### Замечания

Раздел review/findings/escalation.

Внутри:

- итог ревью;
- список замечаний;
- severity;
- рекомендации;
- будущие repair-действия.

### Технические детали

Operator-only depth.

Внутри:

- task timeline;
- planning history;
- context manifests;
- execution traces;
- validation runs;
- escalations.

## Навигационная модель

### Глобальная навигация

Глобальный уровень должен быть очень коротким:

- список проектов;
- активный проект.

### Локальная навигация внутри проекта

Рекомендуемый набор разделов:

- `Обзор`
- `Артефакты`
- `Ход выполнения`
- `Состояние`
- `Замечания`
- `Технические детали`

### Правило навигации

Каждая сущность должна открываться в очевидном месте:

- артефакт -> `Артефакты`
- событие про ревью -> `Замечания`
- шаг route/journey -> `Ход выполнения`
- gap/readiness -> `Состояние`
- trace/context/validation -> `Технические детали`

Frontend не должен “угадывать”, куда вести пользователя.
Backend должен отдавать `detail_view` и `entity_type`.

---

## Паттерн представления деталей

Не все детали должны открываться как полный переход на новую страницу.

### Полный экран / основной view

Использовать для:

- workspace overview;
- чтения большого артефакта;
- полноценного review;
- детального journey.

### Drawer / side sheet

Использовать для:

- карточки события timeline;
- компактной карточки шага;
- короткого state detail;
- технической справки.

### Modal

Использовать только для:

- подтверждения действия;
- короткого ввода;
- рискованных команд.

Нельзя превращать modal в полноценный рабочий экран.

---

## Принципы адаптивности

### Desktop

Основной режим.

Рекомендуемая композиция:

- слева: project rail / список проектов;
- в центре: workspace overview;
- справа: contextual side rail с артефактами и summary.

### Tablet

Переход к двухзонной модели:

- основной контент;
- выдвижная панель деталей.

Journey strip остаётся сверху.
Контекстные блоки переходят вниз по странице.

### Mobile

Один столбец.

Приоритет:

1. Header
2. Journey
3. Situation + Primary Action
4. Timeline
5. Artifacts

Детали должны открываться как bottom sheet или отдельный route.

## Realtime-модель

UI не должен:

- перезагружать весь экран по таймеру;
- повторно запрашивать все данные без необходимости;
- держать giant store “на всякий случай”.

Правильная модель:

- initial load по REST;
- точечные обновления по WebSocket;
- перезапрос только той проекции, которая изменилась.

### Источник realtime

`WS /ws/projects/{project_id}`

### Тип событий

Уже сейчас:

- `snapshot`
- `projection_changed`

UI должен считать WebSocket триггером на **частичную инвалидацию**.
Полную проекцию он перечитывает через REST.

---

## Что потребуется на M10

Для целевого UX текущего `M9` уже достаточно для read-only operator surface.

Но для полноценного пользовательского процесса понадобятся дополнительные серверные модели:

- `interactions`
  - запросы на уточнение;
  - approvals;
  - решения пользователя;
- `repair cycle`
  - findings;
  - repair status;
  - re-review;
- `action availability`
  - почему action доступен или недоступен;
  - нужен ли confirm;
- timeline entries для:
  - clarification requested;
  - clarification answered;
  - repair started;
  - repair finished;
  - accepted risk.

---

## UX-инварианты

1. Главный экран должен давать картину проекта целиком.
2. Операционная лента обязательна и является центральным элементом overview.
3. На экране только один primary action.
4. Все технические детали скрыты глубже.
5. Сначала показывается человеческий смысл, потом внутренняя механика.
6. Domain packs должны быть видны пользователю как логичное расширение процесса, а не как скрытая инженерная надстройка.
7. UI строится поверх независимых серверных проекций, а не поверх giant DTO.
