# Техническое задание: Разработка базового фреймворка платформы E2E-генерации PoV

## 1. Общие сведения и архитектурная парадигма

**Цель:** Разработка масштабируемой платформы для автоматической E2E-генерации артефактов Proof of Concept (ТЗ, исходный код, документация) на основе бизнес-запросов. Первичный домен: Data Science.

**Ключевой принцип:** Платформа нацелена на изоляцию и детерминизацию вызовов агентов для повышения стабильности и предсказуемости. Агенты функционируют как строго типизированные вычислительные узлы для трансформации данных и планирования в рамках заданных контрактов.

**Оркестрация:** Реализуется через двухуровневую архитектуру:
1. **Макро-уровень (Stage-Gate):** Управление проектными фазами.
2. **Микро-уровень (DAG Event Loop):** Асинхронное выполнение графа задач.
---
## 2. Инфраструктурные компоненты
Система состоит из изолированных микросервисов:
1. **Stage-Gate Manager:** Контролирует макро-фазы. Блокирует переход пула задач на следующий этап до успешного завершения глобальной интеграционной проверки.
2. **Task Router (Диспетчер задач):** Асинхронный Event Loop. Опрашивает пул задач, маршрутизирует готовые к выполнению задачи (`Queued`) в профильные бизнес-модули по принципу FIFO.
3. **Template Registry:** Хранилище конфигураций (YAML/JSON) для шаблонов задач. Содержит схемы входов, выходов и правила декомпозиции.
4. **Context Engine:** Подсистема подготовки данных. Собирает контекст для промпта на основе жестких контрактов задачи и/или механизмов поиска.
5. **State & Memory Broker:** Версионируемая база данных (PostgreSQL + S3). Хранит состояния пула задач и файловые артефакты.
6. **Environment Manager:** API для управления средами выполнения. Поддерживает запуск Stateless-контейнеров (Docker) и Stateful-сессий (Jupyter).
7. **Interruption Gateway:** Шина событий для приостановки пайплайна и передачи управления человеку при критических сбоях (исчерпание лимитов самокоррекции, нехватка вводных).
---
## 3. Жизненный цикл и маршрутизация задач
### 3.1. Пайплайн планирования и исполнения
Каждая высокоуровневая задача проходит через строгий алгоритмический конвейер:
1. **Уточнение / Нормирование:** Приведение задачи к стандартизированному формату домена.
2. **Оценка:** Валидация выполнимости. Выявление невозможных задач для раннего сдвига эскалации пользователю.
3. **Декомпозиция:** Разбиение сложных задач на исполнимые шаги на основе шаблонов.
4. **Приоритезация:** Автоматическое выстраивание графа зависимостей (DAG).
5. **Взятие в работу (Пул задач):** Маршрутизация готовых узлов графа в бизнес-модули и песочницы.
### 3.2. Типология шаблонов задач
Шаблоны в `Template Registry` строго классифицируются на три типа:
- **Составной (Composite):** Контейнерная задача. Не передается на исполнение, служит исключительно для статической маршрутизации заранее заданных дочерних подзадач.
- **Исполняемый (Executable / Leaf):** Конечная задача. Выполняется физическим скриптом или LLM-вызовом строго в рамках своего контракта.
- **Динамически декомпозируемый (Dynamic):** Решение о том, выполнить задачу сразу или разбить на подзадачи, принимается «на лету» бизнес-модулем в зависимости от объема контекста и сложности.
### 3.3. Конечный автомат статусов задачи
- **`Blocked`:** Ожидает завершения родительских процессов или зависимостей.
- **`Queued`:** Все зависимости разрешены, готова к взятию в работу Диспетчером.
- **`In_Progress`:** Выполняется внутри бизнес-модуля.
- **`Completed`:** Успешно завершена, артефакт сохранен в БД.
- **`Failed`:** Завершена с ошибкой (лимиты исчерпаны, требуется эскалация).
- **`Waiting_for_Children`:** Задача декомпозирована и ожидает завершения дочерних элементов (механизм Bubble Up).
- **`Obsolete`:** Инвалидирована механизмом перепланирования (откат графа).
---
## 4. Управление памятью и сборка контекста
### 4.1. Контракты зависимостей
Контракт `input_requirements` определяет правила сборки контекста:
- **`Hard` (Жесткое требование):** Артефакт критичен (например, "Схема БД"). При его отсутствии задача переводится в ошибку до вызова LLM.
- **`Soft` (Мягкое требование):** Опциональный артефакт. При наличии подгружается в промпт, при отсутствии игнорируется.
- **`Semantic` (Эвристический поиск):** Запрос к векторной базе (RAG) для поиска релевантных фрагментов по всему проекту.
